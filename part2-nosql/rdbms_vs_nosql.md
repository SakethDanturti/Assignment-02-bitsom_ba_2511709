I would recommend MySQL as the primary database for a patient management system. Healthcare data is highly sensitive and operationally critical, so the system needs strong consistency, reliability, and transactional safety. Patient records, appointments, prescriptions, billing, and test reports often involve multiple related updates that must either all succeed or all fail together. That is exactly where ACID properties are important. MySQL provides atomicity, consistency, isolation, and durability, which helps prevent dangerous issues such as duplicate prescriptions, incomplete billing entries, or mismatched patient records. In a healthcare setting, correctness matters more than flexibility.

From a CAP theorem perspective, most patient management systems should prefer consistency and partition tolerance over eventual consistency. If the network is unstable, it is generally better to delay an update than to allow doctors or staff to see conflicting versions of a patient’s information. MongoDB is useful when data is highly variable or document-based, but its BASE-oriented approach and flexible schema are usually better suited for applications where speed and scalability are more important than strict transaction guarantees.

Yes, my answer would partly change if the startup also needed a fraud detection module. Fraud detection often deals with large, fast-moving, semi-structured data such as device logs, user behaviour, claims patterns, and anomaly signals. For that module, MongoDB could be a strong addition because it handles flexible schemas and high-volume ingestion well. So the best approach would be a hybrid architecture: use MySQL as the system of record for core patient and billing data, and use MongoDB for fraud analytics and event-driven data. In short, MySQL should remain the foundation, while MongoDB can support specialised analytical needs.
